YOLO Pipeline: SVO → trained model → JSONL handoff

How the paper-detector half of the perception stack gets built, trained, and deployed. Written 2026-05-11.

TL;DR. Record SVOs → extract every Nth frame as PNG → upload to Roboflow → label per-color classes → export YOLOv11 dataset → train on a rented cloud GPU (Lambda / Runpod / Vast) → export ONNX → on the Jetson, run a worker that opens each finished SVO segment, runs the model on the left camera frames, writes one JSONL line per detection per frame to <detections_dir>/<segment_id>.jsonl. The SDK does the rest. Resolution normalisation is YOLO's standard letterbox, and the SDK's runtime.detector_image_size handles the bbox rescale.

Where this fits in the topology

The YOLO worker is an independent process. It does not know about circle_processing; it only knows about SVO files in and JSONL files out. The SDK joins YOLO output to ZED depth/pose/plane data by (segment_id, frame_idx).

The contract we have to produce

Recap from perception/zed-positional-measurement/src/zed_positional_measurement/models.py:

// One line per detection, multiple papers per frame = multiple lines.
{
  "frame_idx": 17,                              // local to the segment, 0-based
  "bbox_xyxy": [x1, y1, x2, y2],                // integer pixels
  "label": "paper",                             // semantic label
  "color": "red",                               // paper colour / class
  "confidence": 0.92                            // 0..1
}

One file per recorded SVO segment, written to <paper.external_detections_dir>/<segment_id>.jsonl. The SDK reads it back via SegmentJsonlPaperProvider when paper.mode = "external_boxes".

End-to-end stages

Stage 1 — Capture SVOs (already done)

perception/record_svo.py records HD1200 @ 30 fps to recordings/<name>.svo2. You'll feed those SVOs into stage 2 for training frame extraction, and later into stage 8's inference worker for production runs.

Tip. Vary lighting, distance, angle, and background across recordings. The drone will see papers from oblique angles at unpredictable distances — your training set has to too. One 5-minute static recording is a worse dataset than ten 30-second varied ones.

Stage 2 — Extract training frames (already done)

python3 extract_training_frames.py --svo recordings/flight_01.svo2 --every_n 15

That gives you ~2 fps of PNGs at HD1200 in training_frames/flight_01/. The script uses sl.VIEW.LEFT, so the images are the left ZED camera, which is also what the inference worker will use in stage 8 — train and infer on the same view.

every_nEffective fpsGood for
301 fpslong recordings, you want diversity not duplication
15 (default)2 fpsstandard
56 fpsshort recordings, dense coverage
130 fpsalmost never — adjacent frames are near-duplicates and waste label hours

Stage 3 — Label in Roboflow

upload_to_roboflow.py pushes the PNGs to your Roboflow project. Then you label them in Roboflow Annotate.

Class strategy

The SDK's PaperDetection carries both label and color. The clean way to produce both from a single-output YOLO is to train one class per colour:

Roboflow class nameSDK class_idlabelcolor
paper_red0paperred
paper_blue1paperblue
paper_yellow2paperyellow
…………

The worker (stage 8) then maps class_id → (label, color) using paper.classes in the SDK config. This keeps the model output dense (one detection = one class) and avoids ambiguous multi-task heads.

DECIDE. How many target colours are in the AEAC 2026 spec? That determines class count. If it's ≤ 8 colours we can stay with per-colour classes; if it ever balloons past ~20, switch to one paper class + a separate colour classifier head, but you don't need that now.

Labeling tips that matter for this dataset

  • Box the entire visible paper, not just its centre — YOLO learns the box and we use its centre pixel downstream for the depth sample.
  • Occluded papers: box only the visible portion, mark as the colour you can see. The detector should learn to ignore "I can only see 15% of it."
  • Don't label papers that are blown out / motion-blurred to the point you can't tell the colour. Garbage labels poison the model more than missing labels do.
  • Aim for at least 300–500 boxes per colour for a solid baseline; 1000+ per colour for production-grade.

Stage 4 — Generate the Roboflow dataset version

In Roboflow → Generate → pick:

  • Preprocessing: Resize to 640 × 640 (Fit within → letterbox). This is the standard YOLO input size and what we'll set detector_image_size to.
  • Augmentations (start mild, add more if val accuracy diverges from train):
    • Horizontal flip
    • Brightness ±15%, Exposure ±10% — drone flights have variable lighting
    • Blur up to 1.5px — simulates motion blur
    • Noise up to ~2% — sensor noise robustness
    • Skip vertical flip (papers have an "up") and skip aggressive rotation if you trust your camera orientation
  • Train/Valid/Test split: 70/20/10 is fine.
  • Export format: "YOLOv11" (also works as YOLOv8). You get a zip with data.yaml, train/, valid/, test/.

What to look at after training

  • runs/papers_v1/confusion_matrix.png — if blue is getting predicted as purple a lot, that's a labeling/lighting problem, not a model problem.
  • runs/papers_v1/val_batch*.jpg — eyeball the predictions on validation images. Look for class confusions and missed detections at small sizes.
  • The best checkpoint is at runs/papers_v1/weights/best.pt.

Stage 6 — Resolution normalisation (the part you asked about explicitly)

This is where most people get the bbox coords wrong. Three reference frames:

FrameWhat it is
NativeThe ZED's actual capture, e.g. HD1200 = 1920 × 1200. The SVO segments are recorded at this resolution.
Detector inputThe size your YOLO model expects. Standard is 640 × 640.
Detector outputYOLO returns boxes in detector-input coordinates (0..640).

How the math works

ZED images are 1920 × 1200, which is not square. To feed YOLO a 640×640 input, the standard approach is letterbox:

  1. Scale the longer side (1920) down to 640: scale factor = 640 / 1920 = 0.333.
  2. Apply the same scale to the shorter side: 1200 × 0.333 = 400.
  3. Pad top + bottom with grey to reach 640 height: (640 − 400) / 2 = 120 px of padding on each side.
  4. Run YOLO on the resulting 640×640 image.
  5. For each output box (x1, y1, x2, y2) in YOLO coords, reverse the transform:
    • Subtract the pad: y1 -= 120, y2 -= 120
    • Divide by scale: divide all four values by 0.333 to get back to native pixels

Ultralytics does all of this for you when you use its inference API — boxes come back in native input image coordinates automatically. The stage 8 worker uses that, so we don't write the letterbox math by hand.

Where the SDK does its own scaling

Even though Ultralytics returns native-coord boxes, you can optionally tell the SDK what resolution your YOLO ran at via runtime.detector_image_size. If you set it equal to the native image size (e.g. [1920, 1200]), no rescaling happens. If you set it to the YOLO input size (e.g. [640, 640]), the SDK rescales the boxes from 640-space to native-space using geometry.scale_bbox_xyxy.

Watch out. Decide one approach and stick to it. Recommended: write JSONL in native pixel coords (Ultralytics gives you these for free) and set runtime.detector_image_size to None (the default). Then there is no extra scaling in the SDK and no chance of double-scaling boxes.

Stage 7 — Export the trained model

Export to ONNX so any inference runtime can load it. The flag dynamic=False bakes in the 640×640 input shape, which TensorRT prefers on Jetson.

yolo export model=runs/papers_v1/weights/best.pt format=onnx imgsz=640 dynamic=False simplify=True

You get best.onnx. Copy it to the Jetson (e.g. perception/models/papers_v1.onnx).

TensorRT (optional, for max Jetson speed). Ultralytics can convert ONNX to a TensorRT engine on the Jetson itself: 
yolo export model=papers_v1.onnx format=engine device=0 imgsz=640
The first inference on the Jetson will take a minute (TRT engine build); subsequent runs are 2–3× faster than raw ONNX. You can also just keep the .pt file on the Jetson and let Ultralytics auto-convert at first inference — same end state.

Stage 8 — Build the YOLO worker (Jetson)

This is the new script. It opens each finished SVO segment, runs YOLO on every left-camera frame, and writes JSONL per the contract.

#!/usr/bin/env python3
"""
yolo_worker.py — Run on Jetson. Watches segments/ for new SVO files and emits
one JSONL per segment matching the contract in
zed_positional_measurement/models.py:PaperDetection.from_dict.

Usage:
    python3 yolo_worker.py \\
        --segments-dir runs/<session_id>/segments \\
        --out-dir      runs/<session_id>/yolo \\
        --model        perception/models/papers_v1.pt
"""

import argparse, json, time
from pathlib import Path
import pyzed.sl as sl
from ultralytics import YOLO

# class_id (Roboflow class index) → (label, color) for the SDK
CLASS_MAP = {
    0: ("paper", "red"),
    1: ("paper", "blue"),
    2: ("paper", "yellow"),
    # extend with the rest of your AEAC colours
}

def process_segment(svo_path: Path, out_path: Path, model: YOLO,
                    conf: float = 0.25) -> int:
    """Process one SVO segment → one JSONL file. Returns frames processed."""
    zed = sl.Camera()
    params = sl.InitParameters()
    params.set_from_svo_file(str(svo_path))
    params.svo_real_time_mode = False
    if zed.open(params) != sl.ERROR_CODE.SUCCESS:
        raise RuntimeError(f"failed to open {svo_path}")

    image = sl.Mat()
    local_frame_idx = 0

    out_path.parent.mkdir(parents=True, exist_ok=True)
    with out_path.open("w") as handle:
        while zed.grab() == sl.ERROR_CODE.SUCCESS:
            zed.retrieve_image(image, sl.VIEW.LEFT)
            img_bgr = image.get_data()[:, :, :3]  # BGRA → BGR

            # Ultralytics returns boxes in native image coords. We pass
            # imgsz=640 so internally it letterboxes to 640 then reverses.
            results = model(img_bgr, imgsz=640, conf=conf, verbose=False)

            for box in results[0].boxes:
                cls_id = int(box.cls[0])
                if cls_id not in CLASS_MAP:
                    continue
                label, color = CLASS_MAP[cls_id]
                x1, y1, x2, y2 = (int(round(v)) for v in box.xyxy[0].tolist())
                handle.write(json.dumps({
                    "frame_idx": local_frame_idx,
                    "bbox_xyxy": [x1, y1, x2, y2],
                    "label": label,
                    "color": color,
                    "confidence": float(box.conf[0]),
                }) + "\n")
            local_frame_idx += 1

    zed.close()
    return local_frame_idx


def main():
    ap = argparse.ArgumentParser()
    ap.add_argument("--segments-dir", required=True, type=Path)
    ap.add_argument("--out-dir",      required=True, type=Path)
    ap.add_argument("--model",        required=True, type=Path)
    ap.add_argument("--conf", type=float, default=0.25)
    ap.add_argument("--poll-seconds", type=float, default=2.0,
                    help="how often to scan for new finished segments")
    args = ap.parse_args()

    model = YOLO(str(args.model))
    seen: set[str] = set()

    while True:
        for svo in sorted(args.segments_dir.glob("segment-*.svo2")):
            seg_id = svo.stem  # "segment-00001"
            out_path = args.out_dir / f"{seg_id}.jsonl"
            if seg_id in seen or out_path.exists():
                seen.add(seg_id)
                continue
            # Recorder writes segments sequentially. A segment-NNNNN is
            # "finished" once segment-N+1 exists.
            next_id = f"segment-{int(seg_id.split('-')[1]) + 1:05d}.svo2"
            if not (args.segments_dir / next_id).exists():
                continue
            print(f"[yolo_worker] processing {seg_id}")
            n = process_segment(svo, out_path, model, conf=args.conf)
            print(f"[yolo_worker]   wrote {n} frames → {out_path}")
            seen.add(seg_id)

        time.sleep(args.poll_seconds)


if __name__ == "__main__":
    main()
"Segment is finished" detection. The recorder rolls segments every segment_duration_s (currently 20s). The simplest signal that segment N is closed is that segment N+1 exists. This is what the snippet above uses. If we ever want zero-lag-after-last-segment behaviour, we'll add a sentinel file on session stop.

Decisions you still need to make

  1. Class count. How many target colours does AEAC 2026 require? — drives Roboflow class setup.
  2. YOLO model size. Start with yolo11s. Drop to n if Jetson is fps-bound.
  3. Cloud provider. Runpod 4090 spot for first cut. Lambda A100 if reliability matters more than $.
  4. SVO ↔ training frame budget. Aim for 300–500 labeled boxes per colour minimum, 1000+ for production. every_n=15 from a 5-minute flight gives ~600 frames — at ~3 papers per frame that's enough labels for one colour, so plan recordings accordingly.
  5. Run YOLO on the SVO or live? Above is the simpler design — let the recorder write SVO segments and have the YOLO worker tail the segments directory. If we ever want to skip the SVO disk hop and tee the live camera frames into both the recorder and YOLO, that's a bigger refactor and not needed for the smoke test.

Things to watch on the first hardware run

  • Per-segment YOLO worker wall-time. If a 20s segment takes > 20s to process, YOLO becomes the bottleneck and frame stream falls behind real time.
  • Class confusion in circle_processing.cluster_papers output. If yellow papers are getting clustered as red, the labeling was inconsistent and the cluster colour vote will fail loudly.
  • Detection rate at far range. ZED at HD1200 sees ~5m papers clearly; YOLO will get lossy past that. Compare detection density vs. estimated distance in frames.jsonl for the first flight to calibrate expectations.
  • Frame-index alignment. The first ZED grab() on an SVO is local_frame_idx = 0. The SDK's process_segment uses the same 0-based convention. Sanity-check by opening the first JSONL and confirming detections at frame_idx=0 exist for the first paper-containing frame in the SVO.
