Detector Pipeline: SVO → RF-DETR → JSONL handoff

How the paper-detector half of the perception stack gets built, trained, and deployed. Written 2026-05-11, updated 2026-05-12 to use RF-DETR via Roboflow Inference instead of Ultralytics YOLO.

TL;DR. Record SVOs → extract every Nth frame as PNG → upload to Roboflow → label per-colour classes → train an RF-DETR Small model in Roboflow's hosted training UI (no GPU rental needed) → on the Jetson, run the Roboflow Inference Docker container locally → run a worker that opens each finished SVO segment, calls Inference on the left camera frames, writes one JSONL line per detection per frame to <detections_dir>/<segment_id>.jsonl. The SDK does the rest. Resolution normalisation (letterbox to 640) happens inside Roboflow Inference; the worker sees boxes in native ZED pixel coords and converts Roboflow's (cx, cy, w, h) format to xyxy before writing.

Where this fits in the topology

The detector worker is an independent process. It does not know about circle_processing; it only knows about SVO files in and JSONL files out. The SDK joins detector output to ZED depth/pose/plane data by (segment_id, frame_idx).

The contract we have to produce

Recap from perception/zed-positional-measurement/src/zed_positional_measurement/models.py:

// One line per detection, multiple papers per frame = multiple lines.
{
  "frame_idx": 17,                              // local to the segment, 0-based
  "bbox_xyxy": [x1, y1, x2, y2],                // integer pixels
  "label": "paper",                             // semantic label
  "color": "red",                               // paper colour / class
  "confidence": 0.92                            // 0..1
}

One file per recorded SVO segment, written to <paper.external_detections_dir>/<segment_id>.jsonl. The SDK reads it back via SegmentJsonlPaperProvider when paper.mode = "external_boxes".

End-to-end stages

Stage 1 — Capture SVOs (already done)

perception/record_svo.py records HD1200 @ 30 fps to recordings/<name>.svo2. You'll feed those SVOs into stage 2 for training frame extraction, and later into stage 8's inference worker for production runs.

Tip. Vary lighting, distance, angle, and background across recordings. The drone will see papers from oblique angles at unpredictable distances — your training set has to too. One 5-minute static recording is a worse dataset than ten 30-second varied ones.

Stage 2 — Extract training frames (already done)

python3 extract_training_frames.py --svo recordings/flight_01.svo2 --every_n 15

That gives you ~2 fps of PNGs at HD1200 in training_frames/flight_01/. The script uses sl.VIEW.LEFT, so the images are the left ZED camera, which is also what the inference worker will use in stage 8 — train and infer on the same view.

every_nEffective fpsGood for
301 fpslong recordings, you want diversity not duplication
15 (default)2 fpsstandard

130 fpsalmost never — adjacent frames are near-duplicates and waste label hours

Stage 3 — Label in Roboflow

upload_to_roboflow.py pushes the PNGs to your Roboflow project. Then you label them in Roboflow Annotate.

Class strategy

The SDK's PaperDetection carries both label and color. The clean way to produce both from a single-output detector is to train one class per colour:

Roboflow class nameSDK class_idlabelcolor
paper_red0paperred
paper_blue1paperblue
paper_yellow2paperyellow
…………

The worker (stage 8) then maps class_id → (label, color) using paper.classes in the SDK config. This keeps the model output dense (one detection = one class) and avoids ambiguous multi-task heads.

DECIDE. How many target colours are in the AEAC 2026 spec? That determines class count. If it's ≤ 8 colours we can stay with per-colour classes; if it ever balloons past ~20, switch to one paper class + a separate colour classifier head, but you don't need that now.

Labeling tips that matter for this dataset

  • Box the entire visible paper, not just its centre — the detector learns the box and we use its centre pixel downstream for the depth sample.
  • Occluded papers: box only the visible portion, mark as the colour you can see. The detector should learn to ignore "I can only see 15% of it."
  • Don't label papers that are blown out / motion-blurred to the point you can't tell the colour. Garbage labels poison the model more than missing labels do.
  • Aim for at least 300–500 boxes per colour for a solid baseline; 1000+ per colour for production-grade. RF-DETR is more sample-efficient than YOLO, so the lower end of this range is realistic for a first cut.

Stage 4 — Generate the Roboflow dataset version

In Roboflow → Generate → pick:

  • Preprocessing: Resize to 640 × 640 (Fit within → letterbox). RF-DETR Small uses 640×640 input — same as YOLO, so the letterbox preprocess is identical and the SDK's runtime.detector_image_size story is unchanged.
  • Augmentations (start mild, add more if val accuracy diverges from train):
    • Horizontal flip
    • Brightness ±15%, Exposure ±10% — drone flights have variable lighting
    • Blur up to 1.5px — simulates motion blur
    • Noise up to ~2% — sensor noise robustness
    • Skip vertical flip (papers have an "up") and skip aggressive rotation if you trust your camera orientation
  • Train/Valid/Test split: 70/20/10 is fine.
  • Export: not needed for the hosted RF-DETR training path. Just hit "Generate" — the trained model lives inside Roboflow and is served via the Inference SDK at stage 7. (You can still export a YOLO-format zip from the same generated version if you ever want to retrain locally; it doesn't break anything.)

Stage 5 — Train in Roboflow's hosted UI

From the generated dataset version, click Train. Pick:

  • Architecture: Roboflow RF-DETR (the "Recommended" option).
  • Model Size: Small. If "Show More" exposes Medium and your training credits allow it, Medium will be slightly more accurate but ~2× the latency on Jetson — still inside our 20s-per-segment budget. Start with Small; if precision/recall isn't where you want it, retrain as Medium on the same version.
  • Train from checkpoint: leave default (Roboflow uses its base RF-DETR pretrained weights).

Roboflow handles compute, hyperparameters, augmentation, and the checkpoint selection. No GPU rental, no SSH, no yolo CLI. The trade is you spend training credits instead of cloud-GPU-hours; for a one-shot competition this comes out roughly the same and saves you the setup.

Tip. Roboflow trains in the cloud and notifies when finished. Don't sit and watch — go label more frames, or work on something else. RF-DETR converges relatively quickly; expect the email back within 30–90 min for a ~1k-image dataset.

What to look at after training

Roboflow's training results page shows:

  • mAP@50 and mAP@50:95 — target ≥ 0.85 on mAP@50 for a clean dataset. RF-DETR Small typically hits 0.90+ on small custom datasets with good labels.
  • Per-class precision / recall — if one colour (e.g. blue) is dragging the mean down, that's a labeling-quality or lighting-coverage problem in your dataset, not a model problem. Fix the dataset, regenerate the version, retrain.
  • Confusion matrix — confirms colours aren't being mistaken for each other.
  • "Try it" tab — drag a held-out test image in and visually inspect predictions before committing the model id to the worker.

Once you're happy, the model is identified by <workspace>/<project>/<version> — that's what the worker references at stage 8.

Stage 6 — Resolution normalisation (the part you asked about explicitly)

This is where most people get the bbox coords wrong. Three reference frames:

FrameWhat it is
NativeThe ZED's actual capture, e.g. HD1200 = 1920 × 1200. The SVO segments are recorded at this resolution.
Detector inputThe size RF-DETR Small expects: 640 × 640.
Detector outputRoboflow Inference returns boxes in native input image coordinates — it reverses the letterbox before handing predictions back.

How the math works

ZED images are 1920 × 1200, which is not square. To feed the detector a 640×640 input, the standard approach is letterbox:

  1. Scale the longer side (1920) down to 640: scale factor = 640 / 1920 = 0.333.
  2. Apply the same scale to the shorter side: 1200 × 0.333 = 400.
  3. Pad top + bottom with grey to reach 640 height: (640 − 400) / 2 = 120 px of padding on each side.
  4. Run the detector on the resulting 640×640 image.
  5. For each output box, reverse the transform: subtract the pad, divide by scale, get back to native pixels.

Roboflow Inference handles every step of this transparently. You hand it a full-resolution image (any aspect ratio), it letterboxes internally, runs RF-DETR at 640×640, then reverses the transform — predictions come back in your input image's native pixel coordinates. The stage 8 worker doesn't write any letterbox math by hand.

Box-format conversion the worker DOES need to do

Roboflow's prediction objects expose (x, y, width, height) where (x, y) is the box centre — not xyxy. The SDK's PaperDetection schema expects bbox_xyxy = [x1, y1, x2, y2]. So the worker converts:

x1 = round(pred.x - pred.width  / 2)
y1 = round(pred.y - pred.height / 2)
x2 = round(pred.x + pred.width  / 2)
y2 = round(pred.y + pred.height / 2)

That's the only coordinate math anywhere in the worker.

Where the SDK's runtime.detector_image_size fits in

The SDK can optionally rescale incoming boxes from a detector-space resolution to the native ZED image size. Since Roboflow Inference already returns native-coord boxes, we leave runtime.detector_image_size = null (the default) and the SDK passes boxes through unchanged.

Watch out. Don't double-scale. Roboflow already reverses the letterbox; if you also set detector_image_size = [640, 640] the SDK will rescale 640-space boxes that are already in 1920×1200 space and your detections will land in the wrong pixels (and the wrong depth samples). Leave detector_image_size = null.

Stage 7 — Deploy to the Jetson via Roboflow Inference

RF-DETR is served through Roboflow Inference, an inference server image that runs locally on the Jetson. No ONNX export, no manual TensorRT conversion — the container does both the first time the model is invoked.

One-time Jetson setup

# on the Jetson, with internet access
pip install inference
# the first model call below also pulls the TRT-Jetson runtime container
# if you'd rather pre-stage it:
docker pull roboflow/roboflow-inference-server-trt-jetson

Warm-up: do this once before competition day

The first call to model.infer() on a Jetson does three things, in order, that you do NOT want happening on the flight line:

  1. Pulls the Roboflow Inference container (~3–5 GB) if it isn't already cached.
  2. Downloads the trained RF-DETR weights for your workspace/project/version from Roboflow.
  3. Compiles the TensorRT engine on the Jetson (~1–2 min on Orin AGX). Cached to disk after.

Do one warm-up inference on a single PNG (or one frame of an SVO) while the Jetson has internet. After that the model runs fully offline.

# warm-up, with internet
python3 -c "
import cv2, os
from inference import get_model
model = get_model(model_id='YOUR_WS/circle-corner-detection/3',
                  api_key=os.environ['ROBOFLOW_API_KEY'])
img = cv2.imread('training_frames/flight_01/frame_00000.png')
print(model.infer(img)[0].predictions[:3])
"
Tip. Save the API key in ~/.bashrc as export ROBOFLOW_API_KEY=... on the Jetson, then it's available to every process without re-typing.

Stage 8 — Build the detector worker (Jetson)

This is the new script. It opens each finished SVO segment, runs RF-DETR via Roboflow Inference on every left-camera frame, and writes JSONL per the contract.

#!/usr/bin/env python3
"""
detector_worker.py — Run on Jetson. Watches segments/ for new SVO files and
emits one JSONL per segment matching the contract in
zed_positional_measurement/models.py:PaperDetection.from_dict.

Uses RF-DETR via the Roboflow Inference SDK. The Inference container handles
letterbox + TensorRT compilation transparently; we just hand it BGR frames.

Usage:
    python3 detector_worker.py \\
        --segments-dir runs/<session_id>/segments \\
        --out-dir      runs/<session_id>/detections \\
        --model-id     YOUR_WS/circle-corner-detection/3
"""

import argparse, json, os, time
from pathlib import Path
import pyzed.sl as sl
from inference import get_model

# class_id (Roboflow class index) → (label, color) for the SDK contract
CLASS_MAP = {
    0: ("paper", "red"),
    1: ("paper", "blue"),
    2: ("paper", "yellow"),
    # extend with the rest of your AEAC colours
}

# Roboflow returns class strings (e.g. "paper_red"). Map them to the SDK's
# (label, color) pair. Used as a fallback if `class_id` is missing.
CLASS_NAME_MAP = {
    "paper_red":    ("paper", "red"),
    "paper_blue":   ("paper", "blue"),
    "paper_yellow": ("paper", "yellow"),
}


def process_segment(svo_path: Path, out_path: Path, model,
                    conf: float = 0.25) -> int:
    """Process one SVO segment → one JSONL file. Returns frames processed."""
    zed = sl.Camera()
    params = sl.InitParameters()
    params.set_from_svo_file(str(svo_path))
    params.svo_real_time_mode = False
    if zed.open(params) != sl.ERROR_CODE.SUCCESS:
        raise RuntimeError(f"failed to open {svo_path}")

    image = sl.Mat()
    local_frame_idx = 0

    out_path.parent.mkdir(parents=True, exist_ok=True)
    with out_path.open("w") as handle:
        while zed.grab() == sl.ERROR_CODE.SUCCESS:
            zed.retrieve_image(image, sl.VIEW.LEFT)
            img_bgr = image.get_data()[:, :, :3]  # BGRA → BGR

            # Roboflow Inference letterboxes to 640 internally and returns
            # predictions in the input image's native pixel coordinates.
            result = model.infer(img_bgr, confidence=conf)[0]

            for pred in result.predictions:
                # Prefer class_id, fall back to class name
                cls_id = getattr(pred, "class_id", None)
                if cls_id is not None and int(cls_id) in CLASS_MAP:
                    label, color = CLASS_MAP[int(cls_id)]
                elif pred.class_name in CLASS_NAME_MAP:
                    label, color = CLASS_NAME_MAP[pred.class_name]
                else:
                    continue  # skip unknown classes

                # Roboflow gives (cx, cy, w, h) in native coords. Convert to xyxy.
                cx, cy, w, h = pred.x, pred.y, pred.width, pred.height
                x1 = int(round(cx - w / 2))
                y1 = int(round(cy - h / 2))
                x2 = int(round(cx + w / 2))
                y2 = int(round(cy + h / 2))

                handle.write(json.dumps({
                    "frame_idx": local_frame_idx,
                    "bbox_xyxy": [x1, y1, x2, y2],
                    "label": label,
                    "color": color,
                    "confidence": float(pred.confidence),
                }) + "\n")
            local_frame_idx += 1

    zed.close()
    return local_frame_idx


def main():
    ap = argparse.ArgumentParser()
    ap.add_argument("--segments-dir", required=True, type=Path)
    ap.add_argument("--out-dir",      required=True, type=Path)
    ap.add_argument("--model-id",     required=True,
                    help="Roboflow model id, e.g. workspace/project/version")
    ap.add_argument("--api-key", default=os.environ.get("ROBOFLOW_API_KEY"),
                    help="Roboflow API key (or set ROBOFLOW_API_KEY env)")
    ap.add_argument("--conf", type=float, default=0.25)
    ap.add_argument("--poll-seconds", type=float, default=2.0,
                    help="how often to scan for new finished segments")
    args = ap.parse_args()

    if not args.api_key:
        ap.error("api key required (--api-key or ROBOFLOW_API_KEY env)")

    print("[detector_worker] loading model (may compile TRT engine on first run)…")
    model = get_model(model_id=args.model_id, api_key=args.api_key)
    print("[detector_worker] model ready")

    seen: set[str] = set()
    while True:
        for svo in sorted(args.segments_dir.glob("segment-*.svo2")):
            seg_id = svo.stem  # "segment-00001"
            out_path = args.out_dir / f"{seg_id}.jsonl"
            if seg_id in seen or out_path.exists():
                seen.add(seg_id)
                continue
            # Recorder writes segments sequentially. A segment-NNNNN is
            # "finished" once segment-N+1 exists on disk.
            next_id = f"segment-{int(seg_id.split('-')[1]) + 1:05d}.svo2"
            if not (args.segments_dir / next_id).exists():
                continue
            print(f"[detector_worker] processing {seg_id}")
            n = process_segment(svo, out_path, model, conf=args.conf)
            print(f"[detector_worker]   wrote {n} frames → {out_path}")
            seen.add(seg_id)

        time.sleep(args.poll_seconds)


if __name__ == "__main__":
    main()
"Segment is finished" detection. The recorder rolls segments every segment_duration_s (currently 20s). The simplest signal that segment N is closed is that segment N+1 exists. This is what the snippet above uses. If we ever want zero-lag-after-last-segment behaviour, we'll add a sentinel file on session stop.

Decisions you still need to make

  1. Class count. How many target colours does AEAC 2026 require? — drives the Roboflow class setup.
  2. RF-DETR Small vs Medium. Default to Small. Switch to Medium only if the Small training run lands at < 0.85 mAP@50 and you've already cleaned the labels. Medium roughly doubles Jetson latency but still fits comfortably inside the 20s-per-segment budget.
  3. SVO ↔ training frame budget. Aim for 300–500 labeled boxes per colour minimum, 1000+ for production. every_n=15 from a 5-minute flight gives ~600 frames — at ~3 papers per frame that's enough labels for one colour, so plan recordings accordingly. RF-DETR is sample-efficient enough that the lower end works for a first-cut model.
  4. Run the detector on the SVO or live? Above is the simpler design — let the recorder write SVO segments and have the detector worker tail the segments directory. If we ever want to skip the SVO disk hop and tee the live camera frames into both the recorder and the detector, that's a bigger refactor and not needed for the smoke test.

Things to watch on the first hardware run

  • Per-segment detector wall-time. If a 20s segment takes > 20s to process, the detector becomes the bottleneck and the frame stream falls behind real time. Expected on Orin AGX with RF-DETR Small + 30fps SVO: ~10s per 20s segment.
  • Class confusion in circle_processing.cluster_papers output. If yellow papers are getting clustered as red, the labeling was inconsistent and the cluster colour vote will fail loudly.
  • Detection rate at far range. ZED at HD1200 sees ~5m papers clearly; even a strong model degrades past that. Compare detection density vs. estimated distance in frames.jsonl for the first flight to calibrate expectations.
  • Frame-index alignment. The first ZED grab() on an SVO is local_frame_idx = 0. The SDK's process_segment uses the same 0-based convention. Sanity-check by opening the first JSONL and confirming detections at frame_idx=0 exist for the first paper-containing frame in the SVO.
  • Box-format correctness. Roboflow returns (cx, cy, w, h) — confirm the worker's xyxy conversion lands on the actual paper by overlaying boxes from the first JSONL onto the corresponding SVO frame. If they're shifted, the conversion math is wrong, not the model.
